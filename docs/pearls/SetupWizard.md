# Setup Wizard (BYOM)
## PEARL Alignment Document

This document defines the quality gates for the **Setup Wizard** UI interface (`SetupWizard.jsx`). It follows the **EYE** (Evaluation, Yield, Evolve) stage of the ADDIECRAPEYE lifecycle.

### P - Perspective
*Who is the user in this view, and what is their immediate goal?*
- **User State (Cognitive Load: Medium - Blocked):** The user is the "System Administrator". They are currently blocked and possibly frustrated because the system is offline or booting for the first time.
- **Goal:** To "Plug in their Mind"—selecting and verifying an active inference engine (LM Studio, Ollama, or Custom Endpoint) so the system can boot and route API traffic.

### E - Engineering
*What backend services, APIs, or components drive this view?*
- **Component:** `SetupWizard.jsx` renders conditionally at the root of `App.jsx`.
- **Backend Bindings (Rust):**
  - **`crate::inference_router::get_active_model`**: Traps the user here if this health check fails during global application mount.
  - **`crate::inference_router::configure_backend`**: The POST route (`/api/config/setup`) where the selected agnostic `backend` (LM Studio/llama.cpp/Ollama) is initialized.
  - **`crate::main::AppState` / `crate::character_sheet::init_db`**: Persists changes so the Wizard bypasses on subsequent boots.
- **Dependencies:** 
  - The Rust backend attempts to handshake with the provided configuration. If successful, the wizard receives `res.ok` and unmounts to reveal the App Chrome.

### A - Aesthetic
*What is the visual language, color scheme, and mood?*
- **Palette/Mood:** Technical, electrical "power-up" motif. The color palette deviates from the warm gold of the active OS to a stark, machine-like slate (`#0F172A` background, `#1E293B` container) with bright cerulean accents (`#0284C7` and `#38BDF8`), simulating the cold hardware state before the "Soul" (AI) is injected.
- **Key Elements:** 
  - Full-screen modal overlay locking the interaction space.
  - Radio button options expand and glow blue when selected (`box-shadow: 0 0 10px rgba(56, 189, 248, 0.2)`).
  - The 🚂 emoji grounds the brand, even in setup mode.

### R - Research
*What data informs this design? How does it align with the Fancy Bible?*
- **Agnostic Architecture Proof:** This view is the direct representation of the "Phase 3 Agnostic Architecture" detailed in `TRINITY_FANCY_BIBLE.md`. By forcing the user to connect `crate::inference_router` themselves, it formalizes the "Bring Your Own Mind" (BYOM) requirement, proving Trinity operates as a lightweight thin-client safely decoupled from hardware limits.

### L - Layout (C.R.A.P. Methodology)
*How does this view use Contrast, Repetition, Alignment, and Proximity?*
- **Contrast:** Intentionally feels "cold" vs the rest of the OS. Features a stark slate background (`#0F172A`) with glowing neon blue (`#0284C7`) inputs. This contrasts entirely with the warm gold of the active Iron Road, signaling the "hardware state" before the "soul" (AI) is injected.
- **Repetition:** The 🚂 emoji continues to persist in the header, repeating the core Trinity identity even when the system is offline to maintain brand trust.
- **Alignment:** Fixed vertical centering constraint. The user flow is locked perfectly dead-center into a single linear column to prevent navigation outside the setup scope. 
- **Proximity:** The primary `Save & Connect` button is placed directly beneath the `Backend URL` input, physically linking the connection action to the string required to establish it.
