# Yard Browser (DOM Viewer)
## PEARL Alignment Document

This document defines the quality gates for the **Yard Browser** UI interface (`YardBrowser.jsx` or native Tauri webview window), functioning as the visual testing sandbox within the Yardmaster Toolkit. It follows the **EYE** (Evaluation, Yield, Evolve) stage of the ADDIECRAPEYE lifecycle.

### P - Perspective
*Who is the user in this view, and what is their immediate goal?*
- **User State (Cognitive Load: High - Verification):** The user acts as the "Observer/QA Tester" verifying the agent's work.
- **Goal:** To render the live application, inspect the visual DOM natively, capture screenshots, and visually iterate on GUI adjustments without leaving the Trinity OS.

### E - Engineering
*What backend services, APIs, or components drive this view?*
- **Component:** A dedicated Tauri webview or embedded Chromium instance accessible via the Yardmaster.
- **Backend Bindings (Rust):**
  - **`crate::agent::tool_execution`**: Exposes the `read_browser_page` / DOM scanning tools directly to Pete's agentic loop, allowing the AI to "see" what it built.
  - **`crate::creative::capture_screenshot`**: Integrates with backend headless browser instances (like Puppeteer or Playwright bindings) to return visual artifacts back into the chat context.
- **Dependencies:** 
  - Subscribes to the local dev server (e.g., `http://localhost:5173`) running the user's Daydream engine or web UI.

### A - Aesthetic
*What is the visual language, color scheme, and mood?*
- **Palette/Mood:** Technical Sandbox. Similar to Chrome DevTools. It utilizes a stark, unpadded window frame so it doesn't pollute the target UI's aesthetics.
- **Key Elements:** 
  - Standard browser chrome (URL bar, Refresh, Forward/Back) rendered in the dark Yardmaster theme.
  - An "Agent Eye" overlay indicator (e.g., a glowing red 🔴 recording dot or a scanning laser line) to visually inform the user when Pete is actively reading the DOM or snapping a screenshot.

### R - Research
*What data informs this design? How does it align with the Fancy Bible?*
- **The Visual Verification Loop:** A core weakness of headless LLMs is CSS/UI blindness. The Fancy Bible demands that the system can *Yield* high-quality aesthetic output. By supplying a dedicated DOM Viewer PEARL, we guarantee that the "Exhale" phase natively includes a visual QA step. It allows the system to operate exactly like the Antigravity IDE.

### L - Layout (C.R.A.P. Methodology)
*How does this view use Contrast, Repetition, Alignment, and Proximity?*
- **Contrast:** The stark, dark Yardmaster UI explicitly contrasts against the embedded Tauri Webview, immediately framing the workspace as "the system" viewing "the product".
- **Repetition:** The "Agent Eye" overlay indicator (e.g., a glowing red 🔴 recording dot) pulses repeatedly to visually communicate when the background agent is actively interacting with the DOM.
- **Alignment:** Dynamically docks to the right-hand split of the Yardmaster, adhering to standard IDE preview norms (e.g., VSCode Live Preview or Windsurf webview).
- **Proximity:** Browser controls (Refresh, Inspect, Forward/Back) are grouped tightly on the top bar of the webview container, separate from Trinity's core controls, mimicking standard web browser ergonomics.
