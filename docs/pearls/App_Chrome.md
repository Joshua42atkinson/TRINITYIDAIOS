# App Chrome (Global Shell)
## PEARL Alignment Document

This document defines the quality gates for the **App Chrome (Global Shell)** UI interface, primarily consisting of `App.jsx` and `NavBar.jsx`. It follows the **EYE** (Evaluation, Yield, Evolve) stage of the ADDIECRAPEYE lifecycle.

### P - Perspective
*Who is the user in this view, and what is their immediate goal?*
- **User State (Cognitive Load: Low):** The user acts as the "Passenger" checking the digital train schedule. They are seeking wayfinding and reassurance that the system is stable.
- **Goal:** To effortlessly switch between operating modes (ID · Learning, AI · Fun, OS · Work), check system hardware telemetry, and anchor themselves to their overarching identity.

### E - Engineering
*What backend services, APIs, or components drive this view?*
- **Component:** `App.jsx` manages the core routing logic, conditional rendering block, and the universal `<footer>`. `NavBar.jsx` controls the top menu and dropdowns.
- **Backend Bindings (Rust):**
  - **`crate::main::AppState`**: The global state struct that persists the current `AppMode` (IronRoad, Yardmaster, Express, Demo).
  - **`crate::health::system_health`**: Powers the `/api/health` polling and the server-sent events (SSE) that hydrate the universal footer's telemetry.
  - **`crate::main::change_app_mode`**: The route handler for `POST /api/mode` that toggles between UI modes.
  - **`crate::inference_router::get_active_model`**: Drives the root conditional block; if unhealthy, the UI traps the user in `SetupWizard`.
- **Dependencies:** 
  - Subscribes to `useQuest`, `useBestiary`, `useSSE` (listening to `crate::main::sse_handler`), and `usePearl` hooks.

### A - Aesthetic
*What is the visual language, color scheme, and mood?*
- **Palette/Mood:** Dark Mode elegance (`#090a0f` base background). Golden/Bronze accents (`#CFB991`) to reflect "The Bible" and "Chariot" themes, establishing a premium, esoteric operating system feel.
- **Key Elements:** 
  - The universal top nav uses a blurred backdrop filter (`backdrop-filter: blur(16px)`).
  - The universal footer is compact, featuring system telemetry data and emoji mode toggles (`🚂`, `⚡`, `🔧`).
  - Font relies on modern typography with `Cinzel` serif text for "portfolio/Trinity" brand headers to feel academic and serious.

### R - Research
*What data informs this design? How does it align with the Fancy Bible?*
- **State Machine Alignment:** Acts as the literal "Mirror" of the backend. It visually surfaces the global variables held in memory.
- **Telemetry Integration:** The navigation and footer explicitly expose the `crate::inference_router` state to show the active model, and `crate::conductor_leader::AddiecrapeyePhase` for the current quest phase. No mode exists that cannot be mapped back to the core `AppState`.
- **Chariots Tie-in:** The `chariots` documentation viewer ties directly to the four core markdown files listed in the Bible.

### L - Layout (C.R.A.P. Methodology)
*How does this view use Contrast, Repetition, Alignment, and Proximity?*
- **Contrast:** The translucent `backdrop-filter` of the Top Navbar heavily contrasts against the darker scrolling workspaces beneath it, establishing a persistent, unmoving "ceiling". The Footer remains muted until an active telemetry error occurs, prioritizing workspace content over OS chrome.
- **Repetition:** The specific use of brand emotes (🚂, ⚡, 🔧) is repeated globally, anchoring the user's perception of the specific OS mode regardless of which nested page they arrive on.
- **Alignment:** The global shell enforces a rigid `100vw`/`100vh` spanning layout. Primary navigation elements are rigidly left-aligned, while active user settings (Identity, Theme) are tightly right-aligned to free up dead-center focus for the child views.
- **Proximity:** Related system metrics in the footer (CPU | GPU | active_model) are grouped intimately together in the bottom left, completely isolated from the clickable operating mode toggles bunched in the bottom right corner.
