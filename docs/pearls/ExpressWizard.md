# Express Wizard
## PEARL Alignment Document

This document defines the quality gates for the **Express Wizard** UI interface (`ExpressWizard.jsx`), which serves as a rapid-generation bypass to the guided Iron Road. It follows the **EYE** (Evaluation, Yield, Evolve) stage of the ADDIECRAPEYE lifecycle.

### P - Perspective
*Who is the user in this view, and what is their immediate goal?*
- **User State (Cognitive Load: Low-but-Fast):** The user is an "Advanced Operator / Speedrunner" pressed for time. They do not want Socratic friction; they just need output.
- **Goal:** To rapidly answer 3 distinct questions (Subject, Audience/Goal, Output Format) to instantly compile and export a deployable byproduct (e.g., HTML5 Adventure, Quiz).

### E - Engineering
*What backend services, APIs, or components drive this view?*
- **Component:** `ExpressWizard.jsx` uses a 3-step state machine to build a single comprehensive payload.
- **Backend Bindings (Rust):**
  - **`crate::quests::create_pearl`**: The endpoint (`POST /api/pearl`) that rapidly initializes the active project state.
  - **`crate::character_sheet::update_character`**: Mutates user identity and context rapidly (`POST /api/character`).
  - **`crate::inference_router::dispatch_compile`**: Handles the direct dispatch of output generation, sending the compiled payload context straight to the agnostic LLM backends (e.g., LM Studio).
- **Dependencies:** 
  - Rapidly mutates the backend state sequentially, ultimately hitting the inference router.
  - Connects to export endpoints (`/api/eye/export?format=...`) and preview rendering (`/api/eye/preview`) handled by `crate::journal::export_portfolio` logic on completion.

### A - Aesthetic
*What is the visual language, color scheme, and mood?*
- **Palette/Mood:** Streamlined, high-energy, and utilitarian. Uses the Lightning Bolt (⚡) motif heavily. It strips out the high-fantasy padding in favor of bold grid-tiles and high-contrast cerulean and violet command buttons.
- **Key Elements:** 
  - `express-wizard__header` maintains stable context while the body updates.
  - Final action buttons (Export, Save Project, Preview) use translucent colorful backgrounds (`rgba(139,92,246,0.15)`) over dark mode base colors to cleanly identify primary paths.

### R - Research
*What data informs this design? How does it align with the Fancy Bible?*
- **Bypass Routing Architecture:** Ensures Trinity is not purely a "toy" or "learning tool". It establishes OS-level capabilities for rapid content generation, utilizing the exact same backend engine (`crate::inference_router`) as the Iron Road, but bypassing the conversational friction gates managed by `crate::conductor_leader` completely.

### L - Layout (C.R.A.P. Methodology)
*How does this view use Contrast, Repetition, Alignment, and Proximity?*
- **Contrast:** Strips out high-fantasy UI padding. Utilizes bold, high-contrast cerulean and violet (`rgba(139,92,246,0.15)`) action buttons against the dark slate to scream "Click Me" and speed up user input.
- **Repetition:** The 3-step sequence repeats the exact same tile-grid structure, minimizing the learning curve as the user steps through the wizard.
- **Alignment:** Renders as a massive, dead-centered UI block within the root App Chrome, visually intercepting the user's focus so they cannot be distracted by side navigation.
- **Proximity:** The overarching "Selected Subject" header persists closely above the active Question Grid. This maintains contextual state natively, preventing the user from needing a "Back" button to remember what they are generating.
